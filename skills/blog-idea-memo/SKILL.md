# ブログ案メモ

## 目的

短い着想を過剰に展開せず、あとで記事へ育てられる具体性を保って既存のアイデアメモへ反映する。候補選定では、着手しやすさを明示的な評価軸で比較する。

## 入力

- 必須: 追加・整理するブログ案、または候補選定の依頼
- 任意: 対象メモ、配置先の見出し、残したい用語や比較軸

## 手順

1. リポジトリの指示と構成を確認し、既存のアイデアメモを特定する。
   - `AGENTS.md` や `README.md` があれば先に読む。
   - `blog-ideas.md`、`ideas.md`、`backlog.md` など既存の候補を優先し、新しい管理ファイルを安易に作らない。
2. 依頼の境界を判定する。
   - 短い題材の提示はメモ更新として扱う。
   - 「次に書く記事」の相談は候補選定として扱う。
   - 記事本文、調査、検証、公開まで求められている場合は、このスキルの範囲を越えた作業として分ける。
3. 編集前に対象メモの近接する見出しと箇条書きを読み、現在の言語、粒度、階層を把握する。
4. Git 管理下なら作業ツリーと対象ファイルの差分を確認し、既存変更を現在の作業と区別する。
5. アイデアを追加または整理する。
   - 近いテーマの場所へ、周囲と同じ箇条書きの深さで追加する。
   - タイトルだけでは意図が失われる場合に限り、想定読者、課題、比較軸、検証観点を短いサブ項目で補う。
   - 対比する技術や方式を別記事にする意図がある場合は、別々の案として残し、何を比較するかを明記する。
   - 設定や運用が主題に関係する場合は、導入だけでなく切り替え、確認、失敗時の扱いも候補の観点に含める。
6. 候補選定では、次の軸で比較して一つを推薦する。
   - 短くまとまりやすいか
   - 調査・実行検証の範囲が狭いか
   - 読者の具体的な問題に結び付くか
   - 別の場面でも再利用できるか
   - 検索語が明確か
7. 編集した場合は対象ファイルだけの差分を確認し、意図した箇所と粒度になっているか検証する。

## 出力

- 更新したメモのパスと追加・整理した案の要約
- 候補選定の場合は、一つの推薦と評価軸に沿った短い理由
- 対象差分の検証結果と、残した既存変更があればその区別

## ガードレール

- メモ更新の依頼から記事本文を作り始めない。
- 周囲より詳細すぎる設計、サンプルコード、調査結果をメモへ持ち込まない。
- 対比として分けるよう指定された題材を一つの曖昧な案へ統合しない。
- 対象外の変更や未追跡ファイルを編集、ステージ、削除しない。
- 候補選定を執筆開始の承認と解釈しない。
- 既存メモを特定できない場合は、推測で新設せず候補パスと判断材料を提示する。

## 依頼例

- 「ブログネタ: MyBatis の association と collection」
- 「JPA は対比として別記事にしたい」
- 「このアイデア一覧から、次に短く書ける記事を一つ選んで」
