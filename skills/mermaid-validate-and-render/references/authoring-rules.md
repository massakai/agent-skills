# Authoring Rules

## 安全な基本方針

- 要件を表現できる最小の図から始める
- 高度な記法や混在構文より、最小の成功テンプレートを優先する
- ノードやエッジは少しずつ追加し、失敗箇所を局所化する
- ノード ID は `api_server` や `step_1` のように英数字とアンダースコア中心でシンプルに保つ
- `subgraph` の ID やラベルも複雑にしすぎず、記号だらけの ID を避ける

## ラベル

- ラベルに空白、記号、括弧、スラッシュ、コロンが入る場合は、その図種別で安全な引用形式を優先する
- 角括弧ラベルを使える図種別では、ID は単純に保ち、表示文言をラベル側へ寄せる
  例: `A["User enters email"]`
- ラベルの引用が複雑になるなら、まずラベル文言を短くする

## レイアウトと可読性

- ユーザー指定がなければ、基本は縦方向の流れを優先する
- 長すぎるエッジラベルは避け、必要なら情報をノード側へ移す
- 線が密集した読みにくい図は避け、意味が崩れるなら複数図に分割する
- `classDef` を大量に増やすより、少数のスタイル規則で済ませる

## 検証の原則

- 意味のある変更ごとに `parse` 検証を回す
- `parse` 成功後は、必ず実レンダリングしてから完了扱いにする
- `render` 失敗後の修正は、新しい構文ミスを持ち込みうるため必ず `parse` からやり直す

## 図種別の目安

- `flowchart`: 手順、分岐、サービス間の流れ
- `sequenceDiagram`: 時系列のやり取り
- `classDiagram`: 静的構造と関係
- `stateDiagram-v2`: 状態遷移やライフサイクル
- `erDiagram`: データ構造と多重度
